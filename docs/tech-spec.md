**Version:** 1.0  
**Repository:** https://github.com/JuliaSivridi/WaterPlant_PWA

---

## 1. Overview

Plant Watering Tracker is a single-page Progressive Web App for tracking per-plant watering schedules. The user sees a list of plants, each with a 7-day calendar of circles, and taps circles to toggle watered/not-watered status. The app computes whether each day is "watered", "ok" (not yet due), "due today", or "overdue" purely from stored dates and the plant's configured frequency.

**Key design decisions:**

- **No backend, no account.** All data lives in `localStorage` under the key `waterplant-plants`. The key is namespaced to avoid collisions with other apps hosted on the same `juliasivridi.github.io` origin.
- **No router.** The entire app is a single view. Modal state is managed by a single `modalPlant` variable in `App.jsx` (`undefined` = closed, `null` = create mode, plant object = edit mode).
- **No date library.** All date arithmetic uses native `Date` and `toISOString().slice(0,10)` to produce `YYYY-MM-DD` strings, which sort and compare correctly as plain strings.
- **Build output to `/docs`.** GitHub Pages requires either `/` or `/docs` as the source folder; `docs/` is used so that `_docs/` (dev documentation) is not served.
- **PWA icons are square** (green `#4CAF50` fill), not circular, so they render without white corners on Android home screen.

## 3. Architecture

**Pattern:** Flat component tree with a single custom hook as the data layer. No state management library, no context.

```
localStorage ("waterplant-plants")
        ↕  read on init / write on every state change
  usePlants (hook)
        ↓  plants[], addPlant, updatePlant, deletePlant, toggleWatered, replacePlants
  App.jsx  ─────────────────────────────────────────────────────────────
    │  plants + onNameClick + onDayToggle
    ├── PlantList
    │     └── PlantRow (×N)
    │           └── DayCircle (×7)  ← getDayStatus() called per circle
    │
    ├── PlantModal  (modalPlant !== undefined)
    │
    └── Toast  (importError !== '')
```

**Write path (toggle watered):**
1. User taps a `DayCircle` → `onClick` fires
2. `PlantRow.onDayToggle(date)` → `App.toggleWatered(plant.id, date)`
3. `usePlants.toggleWatered` calls `setPlants(prev → …)` — immutable update
4. React re-renders `PlantRow`; simultaneously `useEffect` fires
5. `localStorage.setItem('waterplant-plants', JSON.stringify(plants))`

**Write path (add/edit plant):**
1. FAB click → `setModalPlant(null)` (create) or name click → `setModalPlant(plant)` (edit)
2. User fills modal, clicks Save → `PlantModal.onSave({name, frequencyValue, frequencyUnit})`
3. `App.handleSave` calls `addPlant` or `updatePlant(id, fields)`
4. `usePlants` calls `setPlants` → `useEffect` → localStorage

**Read path:**
1. `usePlants` initialises `useState(load)` — `load()` reads and JSON-parses localStorage
2. If key absent or JSON invalid, falls back to `[]` (silent catch)
3. `plants` array flows down as props; `PlantRow` calls `getLast7Days()` and `getDayStatus()` on each render

**Error handling:**
- localStorage parse error → silently returns `[]`
- Import file: FileReader error → `'Failed to read file'`; non-array JSON → `'Invalid file format'`; plant missing `id` or `name` → `'Invalid plant data'`; all surface in the red toast at bottom of screen

## 5. Data Model

### Plant

Stored as elements of the `Plant[]` array in localStorage.

| Field | Type | Default | Description |
|---|---|---|---|
| `id` | `string` | `uuidv4()` | UUID v4, generated at creation |
| `name` | `string` | — | Plant display name, trimmed before save |
| `frequencyValue` | `number` | — | Interval value; coerced with `Number()` on save |
| `frequencyUnit` | `'days'` \| `'weeks'` | — | Multiplier unit |
| `wateredDates` | `string[]` | `[]` | Array of `YYYY-MM-DD` ISO date strings |
| `createdAt` | `string` | `new Date().toISOString().slice(0,10)` | Creation date, `YYYY-MM-DD` |

**Invariants:**
- `frequencyValue` is validated as `>= 1` and `<= 365` in the modal before save.
- `wateredDates` is an unordered array; `getLastWatered` sorts a copy with `.slice().sort().at(-1)` to find the most recent. Duplicates cannot exist — `toggleWatered` checks `.includes(dateStr)` before adding.
- Plant order is insertion order (no sorting). No drag-and-drop reorder in v1.
- No soft delete. `deletePlant` removes from the array permanently.

## 6. Storage Schema

### localStorage

| Key | Value type | Description |
|---|---|---|
| `waterplant-plants` | `JSON.stringify(Plant[])` | Full plant array, written on every state change |

**Lifecycle:**
- Read: `usePlants` initialises with `useState(load)` — `load()` is called once at mount.
- Write: `useEffect([plants])` fires after every render where `plants` changed — synchronous `setItem`.
- No migration logic. If the stored data is unparseable, the app silently starts with an empty array.
- **No versioning.** Schema changes between app versions require manual data migration or a fresh import.

### Backup files

Export filename: `watering-backup-YYYY-MM-DD.json` (date is today's date at export time).  
Format: `JSON.stringify(Plant[], null, 2)` — pretty-printed Plant array, identical structure to localStorage.  
Import validation: must be an array; each element must have truthy `id` and `name`.

## 7. Data Backup (Export / Import)

No authentication. No server. Backup is the only way to transfer data between devices.

### Export flow
1. User opens menu (⋮) → clicks "Export data"
2. `exportData(plants)` called with current `plants` array
3. `Blob` created with `application/json` MIME type
4. Object URL created, injected into a hidden `<a>` with `download` attribute, `.click()` triggered
5. Object URL revoked immediately after click

### Import flow
1. User opens menu (⋮) → clicks "Import data"
2. Hidden `<input type="file" accept=".json">` programmatically clicked via `fileInputRef.current.click()`
3. `FileReader.readAsText(file)` reads the file
4. On load: JSON.parse → array check → each element `.id`/`.name` check
5. On success: `replacePlants(data)` calls `setPlants(data)` — fully replaces current plant list
6. On error: `importError` state set → red toast rendered
7. **No confirmation dialog before overwrite** (replaces immediately on valid file)
8. `e.target.value = ''` reset allows re-importing the same file

### PlantModal

```jsx
<PlantModal
  plant={Plant | null}   // null = create mode
  onSave={({ name, frequencyValue, frequencyUnit }) => {}}
  onDelete={() => {}}    // only called from confirm-delete sub-dialog
  onClose={() => {}}
```

**Internal state:**

| Variable | Type | Initial | Description |
|---|---|---|---|
| `name` | `string` | `''` | Populated from `plant.name` in edit mode via `useEffect` |

| `freqUnit` | `'days'\|'weeks'` | `'days'` | Populated from `plant.frequencyUnit` in edit mode |

**Form fields:**
- Name: text input, placeholder `"e.g. Ficus"`, `autoFocus`, border 1.5px; focus border `#4CAF50`, error border `#E53935`
- Frequency: number input (width 88px, min 1, max 365) + toggle buttons "days" / "weeks"
  - Toggle: border 1.5px `#E0E0E0`, border-radius 12px; active button background `#4CAF50`, text `#fff`

**Action buttons (bottom of modal):**
- Edit mode: "Delete" button (left, `#E53935` text) → sets `confirmDelete = true`
- Right side always: "Cancel" (`#4CAF50` text) + "Save" (filled `#4CAF50`)

**Delete confirmation (when `confirmDelete === true`):**

## 12. Key Algorithms

### getDayStatus

```
function getDayStatus(plant, dateStr):
  if dateStr in plant.wateredDates → return "watered"   // priority 1
  next = getNextWatering(plant)
  if next is null → return "due"                         // no watering history
  if dateStr < next  → return "ok"
  if dateStr === next → return "due"
  return "overdue"                                       // dateStr > next
```

### getNextWatering

```
function getNextWatering(plant):
  last = getLastWatered(plant.wateredDates)
  if last is null → return null
  days = frequencyUnit === 'weeks' ? frequencyValue * 7 : frequencyValue
  date = new Date(last)
  date.setDate(date.getDate() + days)
  return date.toISOString().slice(0, 10)   // YYYY-MM-DD
```

### getLastWatered

```
function getLastWatered(wateredDates):
  if empty → return null
  return wateredDates.slice().sort().at(-1)
  // ISO strings sort lexicographically = chronologically
  // .slice() prevents mutation of original array
```

### getLast7Days

```
function getLast7Days():
  result = []
  for i from 6 down to 0:
    d = new Date()
    d.setDate(d.getDate() - i)
    result.push(d.toISOString().slice(0, 10))
  return result   // [today-6, today-5, ..., today]
```

**Important:** `getLast7Days()` uses the local device clock with no timezone normalisation. `toISOString()` returns UTC. If the device is in a UTC-offset timezone, dates near midnight may shift by ±1 day relative to the user's local date. This is a known limitation of v1.
